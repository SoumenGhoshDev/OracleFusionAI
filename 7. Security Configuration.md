# 🔐 Security Configuration for Oracle AI Agent Studio

## 📋 Overview

Configuring security for Oracle AI Agent Studio involves the standard considerations of **data security** and **segregation of duties**, along with some universal configurations across product areas and data.

![Security Configuration Overview](Security%20Config%20Overview.jpg)

The configuration journey covers these main areas:

| # | Area | Description |
|---|------|-------------|
| 1 | **Profile Option** | Enable the Security Console session to handle permission groups |
| 2 | **Scheduled Processes** | Synchronize security configurations into the application |
| 3 | **Admin/Creator Roles** | Configure roles for users who will develop, promote, deploy, and launch AI agents |
| 4 | **End User Roles** | Fewer roles needed for users who only consume AI Agent functionality |
| 5 | **Pillar-Based Access** | Control which pillars (HCM, SCM, PRC, CX) a user can access — including templates and out-of-the-box agents |

---

## ⚙️ Prerequisites (Before You Start)

### 1️⃣ Enable the Profile Option

After upgrading to **25C**, a profile option must be set to enable permission group features in the Security Console.

![Profile Option Settings](Profile%20Option%20Settings.jpg)

**What it does:**

- Adds a new **"Enable Permission Groups"** checkbox in the Security Console.
- This checkbox is required to see the **Roles and Permission Groups** tab when creating or editing custom roles.

> 💡 **Why a profile option?** Oracle delivers this as an opt-in feature so customers can evaluate and uptake it on their own schedule — it is **not** enabled across the fleet by default.

> ⚠️ **Troubleshooting:** If you cannot see the "Enable Permission Groups" checkbox in Security Console, the profile option has not been set.

---

### 2️⃣ Run ESS (Enterprise Scheduled Service) Jobs

![Import User and Role Security Data](Import%20User.jpg)

Two critical jobs must be run:

| Job | When to Run | Frequency |
|-----|-------------|-----------|
| **Import Resource Application Security Data** | Run **once** after the 25C upgrade | One-time (caches new Oracle-seeded artifacts so they become usable in Security Console) |
| **Import User and Role Application Security Data** | Run on a regular schedule | Weekly / every 14 days (most customers already have this scheduled) |

> ⚠️ If Oracle seeds new security artifacts (e.g., 10 new items), they won't be available in Security Console until the first job is run.

---

## 🏛️ Pillar-Specific Admin Roles

Oracle AI Agent Studio separates access by **pillars**. The templates available to a user depend on which pillar roles they hold. Below are configurations for each pillar.

---

### 🟦 Admin Role for HCM (Human Capital Management)

There are **two paths** depending on whether the user already has the HCM Administrator Role:

![Admin Role for HCM](Admin%20Role%20for%20HCM.jpg)

#### ✅ Path A — User Already Has HCM Administrator Role

1. **Create a custom role** (e.g., `COE AI Agent Admin for HCM`).
2. On the first train stop, **enable permission groups** — click the checkbox, confirm the pop-up, and click "Enable Permission Groups" again.
3. Assign the duty role:
   - **Roles and Privileges** tab → assign the standard duty role
   - **Roles and Permission Groups** tab → assign **GenAI HCM Administrator Duty**
4. Grant this custom role to the user.

#### 🔧 Path B — User Does NOT Have HCM Administrator Role

If you don't want to make agent designers full HCM administrators (to avoid granting access to unrelated tasks):

1. Create a custom role (same steps as above).
2. Assign **two** duty roles:
   - **GenAI HCM Administrator Duty**
   - **HCM Intelligent Agent Management Duty** *(new duty — already included in the HCM Admin role, but needed explicitly here)*

> 💡 This gives **granular control** — users can design agents without being full HCM administrators.

#### 👁️ What HCM Access Looks Like

- The user can access **HCM family** templates and everything labeled **Common**.
- **No access** to Supply Chain, CX, Procurement, or other pillar families.
- The user **can design and build agents** but **cannot run/test** them without the Explorer Role (covered later).

---

### 🟩 Admin Role for SCM (Supply Chain Management)

SCM takes a **different approach** than HCM — simpler out of the box.

![Admin Role for SCM](Admin%20Role%20for%20SCM.jpg)

#### ✅ Path A — User Already Has Supply Chain Administrator Role

- Simply **enable permission groups** on that role.
- **No need** to create a custom role.
- **No need** to explicitly add a duty role.
- The Supply Chain team has already coded all required access into the delivered Supply Chain Administrator Role.

#### 🔧 Path B — User Does NOT Have Supply Chain Administrator Role

1. Create a custom role.
2. Assign **three duty roles** to it (as shown in the screenshot).

> 🔑 **Key Difference from HCM:** For SCM, if you already have the admin role, you're ready to go with just the permission group enabled. HCM is stricter — requiring a custom role to prevent accidental broad access.

---

### 🟨 Procurement (PRC)

The configuration for Procurement follows a **similar pattern** to HCM and SCM. Refer to the pillar-specific duty roles (e.g., **GenAI Procurement Administrator Duty**) and apply the same custom role approach.

---

## 🌐 Admin Role for All Pillars (Superuser Access)

For users working on **multi-pillar deals** or building agents that span across HCM, SCM, and other product areas, you need **all-pillar access**.

![Admin Role for All Pillars](Admin%20Role%20for%20All%20Pillars'.jpg)

There are **two paths**:

### ✅ Path A — User Has Application Implementation Consultant Role

1. **Enable permission groups** in a custom role.
2. Assign **all pillar-specific duty roles**:

| Duty Role | Pillar |
|-----------|--------|
| GenAI HCM Administrator Duty | HCM |
| GenAI Supply Chain Administrator Duty | SCM |
| GenAI Procurement Administrator Duty | PRC |
| GenAI CX Administrator Duty | CX |

### 🔧 Path B — Create a Custom Role from Scratch

1. Create a new custom role.
2. Assign access to the desired pillars by including the relevant duty roles from the table above.
3. Include the **Explorer Role** (covered next) — you decide whether to bundle it into this admin role or keep it separate.

---

## 🔍 Explorer Role (Runtime Access)

> ⚠️ **Critical:** Everything above enables you to **design and build** agents. To actually **run and test** agents, you need the Explorer Role.

### 📝 What's Required

Ensure the following are configured:

| Requirement | Where to Configure |
|-------------|-------------------|
| **Redwood UI enabled** | Should already be on Redwood |
| **Permission Groups enabled** | As covered in prerequisites |
| **Access Intelligent Agent Chat** privilege | Add on the **Function Security Policies** train stop |
| **GenAI Agent Runtime Duty** | Add on the **Role Hierarchy → Permission Groups** tab |

> 💡 These two items (the privilege + the runtime duty) are the **minimum access** needed to grant any employee the ability to test and interact with agents you've built.

### 👁️ What Explorer Access Looks Like

- Under **Me → Agent Studio**, the user can see and interact with agents they have access to.

### 🛠️ Common Troubleshooting — Custom Role Not Showing in Security Tab

When defining an agent, the last tab is **Security**. If your custom role doesn't appear in the LOV (List of Values):

- The role **must** have the **GenAI Agent Runtime Duty** assigned.
- Only roles with this runtime duty will show up in the agent's Security tab dropdown.

> 🛠️ **Fix:** Add the GenAI Agent Runtime Duty to your custom role → the role will then appear in the LOV.

## 🧭 Key Takeaways

1. **Always start with the profile option** and ESS jobs — these are non-negotiable prerequisites.
2. **Pillar teams may differ** in how they deliver security — SCM is more permissive out of the box; HCM is stricter.
3. **Custom roles are your friend** — they isolate AI Agent Studio changes from your existing security model.
4. **Two layers of access exist:**
   - **Design-time** → Admin/duty roles let you build agents
   - **Runtime** → Explorer role (privilege + runtime duty) lets you run and test agents
5. **Don't forget the Explorer Role** — even full administrators cannot test agents without it.
6. **GenAI Agent Runtime Duty** is the gatekeeper — roles without it won't appear in the agent Security tab.
