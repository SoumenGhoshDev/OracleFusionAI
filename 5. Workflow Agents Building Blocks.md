## 🔷 Node Types at a Glance

There are five families of nodes. Here's what each one brings to the table:

| Node Type | Think of it as... | What it does |
|-----------|-------------------|--------------|
| **AI-Driven Nodes** | The *brain* | Calls LLMs, runs agents & sub-agents, retrieves knowledge via RAG |
| **Data Nodes** | The *hands* | Reads/writes business data, connects to APIs, parses documents |
| **Logic Nodes** | The *calculator* | Runs code, shapes data, validates schemas between steps |
| **Workflow Control Nodes** | The *traffic controller* | Routes, loops, waits for events, and pauses for human approval |
| **Communication Nodes** | The *messenger* | Sends emails, notifies stakeholders, generates documents |

---

## 🤖 AI-Driven Nodes

These are the intelligence layer of your workflow — they understand, decide, and generate.

> **Key Capabilities:**
> - Interpret intent, extract entities, classify, summarize, and generate structured outputs
> - Invoke specialist agents or sub-flows to solve parts of the task
> - Ground decisions in retrieved knowledge (policy docs, runbooks)

### 🧠 LLM Node

At the most basic level, an LLM is itself a node. You're simply making a call to a language model — and you can **swap models anytime**. Need GPT today and something else tomorrow? Just switch it out.

### 🕵️ Agent Node

An agent is a self-contained node that can reason and act. But here's where it gets powerful: agents can contain **sub-agent teams**, creating a nesting structure — like Russian dolls. You can compose agents within agents for complex, multi-step orchestration.

### 🔄 Workflow Node

An entire workflow can be plugged in as a single node inside another agent. This means you can **reuse existing workflows** as building blocks — calling them as part of a sequential or parallel execution path.

### 📄 RAG / Document Tool Node

The familiar RAG (Retrieval-Augmented Generation) tool lets you ground your agent's responses in real documents. Under the hood, there's advanced parsing technology that can handle **virtually any document type** with high accuracy. You can even define extraction schemas on the fly using visual tools.

## 🗄️ Data Nodes

Data nodes are the bridge between your agents and the real world — they fetch, store, and transform enterprise data.

> **Key Capabilities:**
> - Read/write business objects (Fusion) and call external services
> - Process documents and extract structured data
> - Persist and retrieve semantic state for continuity

### 📊 Standard Business Object Functions

All the business object functions you already know and use are available here — no learning curve. They just work inside workflows.

### 🌐 External REST Connections

Need data from an outside system? Connect to any **external REST API** to pull or push information as part of your flow.

### 🔧 Tools & MCP

A variety of tools can be wired into workflows. Notably, **MCP (Model Context Protocol)** is now available as a tool — giving agents a standardized way to access context and services.

### 🔍 Vector Read & Write

Think of this as the agent's **short-term working memory**, not a massive vector database. It lets agents semantically index a set of records on the fly and quickly retrieve what's relevant — like asking, *"Find me something similar to this."* It's built for in-workflow decision-making, not for storing thousands of items.

### 📑 Document Processing & Structured Data Extraction

Feed in documents, pull out structured data. Combine this with external service calls for even richer processing pipelines.

---

## ⚙️ Logic Nodes

Logic nodes are where you write the "glue code" that ties everything together.

> **Key Capabilities:**
> - Normalize and shape payloads between steps
> - Compute thresholds, format requests, and validate schemas
> - Store intermediate results and flags for later routing

You can run **actual code** directly within the workflow — giving you full programmatic flexibility. Need to transform a data payload before passing it to the next agent? Need to validate a schema or compute a threshold? This is where you do it. And the capabilities here are **growing with every release**.

---

## 🔀 Workflow Control Nodes

These nodes are the traffic signals of your workflow — they decide what runs, when, and in what order.

> **Key Capabilities:**
> - Control execution shape: route, loop, run concurrently, wait for events
> - Bring humans into the picture for high-risk actions
> - Support long-running, multi-stage enterprise processes
> - Allow processes to wait before proceeding to next steps

### 🔁 Standard Programming Constructs

You get all the familiar control structures:

- **If/Then Conditions** — branch your workflow based on any logic
- **For Loops** — iterate over a collection of items
- **While Loops** — keep going until a condition is no longer true

Bottom line: if you can think it up, you can program it.

### ✋ Human Approval Gates

Sometimes you need a person to say *"Go ahead."* The workflow simply **pauses and waits** — no timeout, no rush. Once the human approves, execution picks right back up. More on this in the [Human-in-the-Loop](#-human-in-the-loop) section.

## 📧 Communication Nodes

Workflows don't operate in a vacuum — they need to talk to people. Communication nodes handle that.

> **Key Capabilities:**
> - Notify users about status, exceptions, and outcomes
> - Keep humans informed without requiring them to open the workflow

### ✉️ Email Composition & Sending

Your workflow can **compose, preview, and send emails** as part of its execution. Common scenarios include:

- Processing HR documents and emailing employees or managers
- Sending status updates to vendors, suppliers, or any stakeholder

The email is generated automatically, but you can always review and edit it before sending.

### 📝 Document Generation *(Coming Soon)*

On the roadmap — the ability to generate rich output artifacts:

- **PowerPoint** presentations
- **PDF** reports
- **Various document formats**
- **Podcasts** (further out) — auto-generated audio summaries are becoming a popular use case

## 🏛️ Real-World Use Case — When You Need 100% Accuracy

> What happens when "pretty good" isn't good enough? Here's a pattern that combines LLM intelligence with code-level reliability.

### ❓ The Problem

Imagine a government benefits system. Policy documents run **dozens of pages**, packed with eligibility clauses for different classes of people. Traditionally, these are handled by **rules engines** — hand-coded logic that checks each clause. The problem? They're **brittle**. Change one paragraph in the policy, and you may need to rewrite the entire codebase.

### 🚫 Why RAG Alone Falls Short

Your first instinct might be: *"Just throw it into RAG!"* Feed the document in, ask natural language questions, and let the LLM figure it out.

It works — *mostly*. But "mostly" is the problem:

- **95% accuracy?** Not acceptable when someone's benefits depend on it.
- **98% accuracy?** Still not enough.
- **100% accuracy is the only target** — zero variation, zero variability.

Extensive experiments across complex policy documents from around the world confirmed this gap.

### ✅ The Solution — LLM-Generated Deterministic Code

Here's the clever part: instead of using the LLM at runtime, use it **once** to generate the rules as code.

1. Feed the policy document to an LLM
2. The LLM generates **Python code** that faithfully represents every clause and rule
3. A human does a **one-time review**, running the code through test cases
4. Once verified — you're golden. The code runs deterministically from that point forward.

### 🎯 Why This Works So Well

| Benefit | What it means for you |
|---------|----------------------|
| **100% Determinism** | The code runs identically every single time — no surprises |
| **100% Accuracy** | Verified by test cases, not probabilistic like RAG |
| **Speed** | Pure code execution — no LLM call at runtime, so it's fast |
| **Flexibility** | Policy changed? Regenerate the code. No need to rewrite from scratch |
| **Automation** | Once approved, the process runs autonomously |
| **Smart LLM Usage** | The LLM does the heavy lifting once; after that, it steps aside |

### 🔑 Key Takeaway

> You don't have to choose between **LLM intelligence** and **code reliability**. Generate deterministic code with an LLM, verify it once, and get the best of both worlds.

## 🧑‍💼 Human-in-the-Loop

Not everything should be fully autonomous. The best workflows know **when to pause and ask a human**.

### 🔐 For Code Generation & Approvals

When an LLM generates code (like in the policy document use case above), you'll want a human to sign off. They review test cases, confirm the output is correct, and give the green light. After that? The code runs on its own — no human needed, because it's been properly validated.

### 📋 For Daily Document Processing

Consider a workflow that processes invoices to create purchase orders. For **high-value transactions**, you want a human to review before anything is committed. The workflow simply **pauses and waits** for approval — no timeout pressure.

Meanwhile, the system may have already processed **thousands of documents** overnight in minutes. Work that used to take days of manual effort (and was error-prone) is now handled automatically, with humans stepping in only where it matters most.
