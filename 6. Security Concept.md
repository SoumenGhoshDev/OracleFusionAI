# 🔐 Security Concept: Accessing Oracle AI Agent Studio

## 📋 Overview

Getting access to Oracle AI Agent Studio requires careful consideration of **two distinct user groups**:

1. **Administrators / Developers** — who build and manage AI Agent teams
2. **End Users** — who interact with AI Agent functionality in their daily workflows

> **Key Question:** How do administrators and users get access to AI Agent Studio?

---

## ✅ Access Requirements at a Glance

### 🛠️ Administrators & Developers

- Must be granted **roles** that allow them to develop AI Agent teams — including agent tools, agents, and related components.
- Need the ability to build on a **test instance**, then use **Functional Setup Manager** (export/import) to promote components to production.
- Choose **where to deploy** AI Agent teams — primarily via **journeys** that users interact with in their daily navigation, work areas, and workflows.
- Configuration aligns with **pillar-based segregation of duty** for teams of administrators and end users.

### 🔍 The Explorer Role

> **Important:** The ability to *find and use* agent teams must be **explicitly assigned** via the **Explorer Role** — even for administrators.

| User Type | Admin/Developer Roles | Explorer Role |
|---|---|---|
| Administrator / Developer | ✅ Required | ✅ Required |
| End User | ❌ Not needed | ✅ Required |

---

## 💡 Practical Examples

### 🅰️ Example A — Pillar-Specific Developer Access (SCM)

To grant a user **developer and use** capabilities for **SCM AI Agent components only**:

- Assign the **Supply Chain Application Administrator** job role.
- Enable the associated **permission groups**.
- The user gains access to AI Agent Studio — but **only SCM AI Agent templates** are available.

### 🅱️ Example B — End User Access

An end user who **does not** need to develop AI Agents still requires:

- A security role containing the **Access Intelligent Agent Chat** privilege.
- This enables them to use any AI Agent teams available in their work areas and workflows.

> **Note:** Security configurations are highly customized per implementation.

---

## 🆕 New Security Roles After 25C Upgrade

Post-upgrade, Oracle introduces new **administrator roles** including duty roles and privileges that are **pillar-specific**:

| Pillar | Area |
|---|---|
| **HCM** | Human Capital Management |
| **SCM** | Supply Chain Management |
| **PRC** | Procurement |

**Key points:**

- Create **custom job roles** to bundle these new duty roles and privileges based on your organizational needs — this enables **access by pillar**.
- Combine all pillar access into an **All-Pillar Administrator Role** for full access across all AI Agent templates.
- End users need **only the Explorer Role** to use AI Agent functionality.
- Developers and administrators **also need the Explorer Role** for debugging and validation.

---

## 🧭 Navigating to AI Agent Studio

As an administrator or developer:

1. Go to the **Home Page**.
2. Open the **Tools** menu.
3. **AI Agent Studio** appears as a menu item — typically in the middle of the top row.

---

## 🐛 Debugging & Validation

Administrators and developers need to **run agent teams** they build to debug (test and validate) them. This requires:

- The **administrative roles** (for building), **plus**
- The **Explorer Role** (for running and interacting with agents)

> **Without the Explorer Role**, attempting to interact with an agent will result in an *"Assistant is unavailable"* message instead of the expected debug messages and AI Agent responses.
